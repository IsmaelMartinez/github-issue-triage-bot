# Repository Strategist — Roadmap

Date: 2026-03-04 (updated 2026-03-18)

## Where We Are (updated 2026-03-18)

The project is evolving from a doc-grounded issue triage bot into a repository strategist: a GitHub App that maintains institutional memory for software projects and uses it to provide strategic intelligence. The triage pipeline remains as one capability among several, but the value proposition shifts from "responds to issues" to "understands your project's trajectory and advises on it."

The triage pipeline is production-grade and running on teams-for-linux in Stage A (invisible helper mode). All output routes through shadow repos where only the maintainer sees it. The pipeline was simplified in a lean bot pivot: Phases 3 (duplicate detection) and 4b (misclassification) were removed in favour of GitHub native tooling. Two tracks remain active: bug triage (`[Triage]` shadow issues, `lgtm`/`reject`) and enhancement context briefs (`[Research]` shadow issues, `research`/`use as context`/`reject`).

The vector store contains 1,362 issues, 18 troubleshooting docs, 31 ADR/research/roadmap documents, 24 Electron v39 release notes, and 329 Electron v39 GitHub issues. Phase 2 uses per-category relevance thresholds (troubleshooting: 70%, upstream_issue: 45%, with defaults per document type). Dashboard v2 is live at `/dashboard` on Cloud Run. Health monitoring (Q3) and feedback tracking (F1) are implemented.

A competitive analysis confirmed that generic triage (labeling, deduplication, routing) is a commodity. GitHub Agentic Workflows (technical preview since February 2026) will further commoditise operational chores like doc sync, test coverage, and CI failure triage. The differentiator is institutional memory with strategic reasoning: connecting decisions, trends, upstream changes, and issue patterns into recommendations that no stateless agent can produce. See `docs/plans/2026-03-18-repository-strategist-design.md` for the full design.

## What This Roadmap Covers

Four streams, running in parallel:

1. Feedback infrastructure — measuring whether the bot is actually useful
2. Quality evolution — making the bot better based on what we measure
3. Communication — helping users understand what the bot does and trust it
4. Strategic intelligence — proactive pattern detection, roadmap suggestions, and project health briefings

Streams 1-3 focus on improving the existing triage pipeline. Stream 4 adds entirely new capabilities. They don't conflict — F3 (learnings) feeds into Stream 4's accumulated learning, and Q1/Q2 (quality loops) apply to both triage and synthesis outputs. Stream 4 is detailed in `docs/plans/2026-03-18-repository-strategist-design.md` with implementation tasks in `docs/plans/2026-03-18-repository-strategist-implementation.md`.

## Stream 1: Feedback Infrastructure

### Phase F1: Baseline Metrics and @mention Processing [DONE]

Implemented in PR #65 (2026-03-17). The bot now detects when users edit their issue to fill in sections that Phase 1 flagged as missing (via `issues.edited` webhook with `changes.body.from`), and captures `@ismael-triage-bot` mentions as qualitative feedback signals. Both are stored in the `feedback_signals` table (migration 010) and surfaced on the dashboard as a Phase 1 fill rate health card and recent feedback panel. See `docs/plans/2026-03-16-phase-f1-feedback-tracking.md` for the full implementation plan.

Metrics to track, by phase:

- Phase 1 (missing info): Did the user edit their issue after the bot commented? This is the strongest signal — if users actually fill in the missing sections, the prompt worked.
- Phase 2 (troubleshooting): Did the user close the issue without further maintainer intervention? Self-service resolution rate.
- Enhancement agent: Quality scores from the holdout judge. Human override rate (approved vs revised vs rejected). Time from research posted to approval.

Implementation: extend `cmd/sync-reactions` to also check issue edits and state changes after bot comments. Store results in a new `feedback_signals` table. Surface in dashboard.

Files:
- `migrations/007_feedback_signals.sql` (006 is now taken by triage_results)
- `internal/store/feedback.go` — queries for tracking feedback signals
- `cmd/sync-reactions/main.go` — extend to collect feedback signals

### Phase F2: Lightweight Active Feedback

Prerequisite: silent mode selectively disabled for at least one phase. This phase only makes sense once the bot is posting comments again.

Add a small footer to the bot's triage comments inviting feedback. Not a survey — a single line with a link to a pre-filled issue template. The footer reads something like: "Was this helpful? React with a thumbs-up, or [let us know how to improve](link-to-feedback-template)." This follows GitHub Copilot's two-tier pattern: quick signal (one click), optional depth (issue template). The feedback issue template has structured fields: which issue, which section of the comment, what was wrong or missing.

Files:
- `.github/ISSUE_TEMPLATE/bot-feedback.yml` — feedback issue template
- `internal/comment/builder.go` — append feedback footer

### Phase F3: Learnings System (CodeRabbit Pattern)

When a maintainer corrects the bot's output — closes an issue the bot didn't flag as duplicate, changes labels the bot said were correct, or rejects agent research with specific feedback — store those corrections as natural-language "learnings" linked to the repository. Future triage runs query relevant learnings and include them as additional context in the LLM prompt.

This is the highest-value feedback mechanism identified in the research. It turns negative feedback into direct improvement rather than just a metric.

Files:
- `migrations/008_learnings.sql` — learnings table (repo, category, learning_text, source_issue, embedding)
- `internal/store/learnings.go` — CRUD + vector search for relevant learnings
- `internal/phases/` — inject relevant learnings into phase prompts

## Stream 2: Quality Evolution

### Phase Q1: Evaluator-Optimizer Revision Loop

Currently the quality judge scores research and logs the result. If the score is low, a human still has to request a revision. Add one automatic revision attempt: if the judge scores < 70, feed the dimension-level feedback back to the research synthesizer and re-generate, capped at one retry. This reduces the number of low-quality outputs that reach human reviewers.

The Google Cloud architecture guide documents this as the "Review and Critique" pattern. The key is that the judge's feedback is specific (which dimensions scored low and why), not just a number.

Prerequisite: Phase F1 metrics showing revision request rate from humans. If humans rarely revise, the loop isn't needed.

Files:
- `internal/agent/judge.go` — add per-dimension feedback strings to QualityScore
- `internal/agent/handler.go` — add revision loop between judge and comment posting

### Phase Q2: Progressive Threshold Adjustment

The confidence-based approval mode uses a static threshold (85/100). Once we have 50+ scored research documents, start tracking per-repo quality distributions. If a repo's average quality score is consistently above 90 and human override rate is below 10%, recommend lowering the approval threshold. If override rate spikes, recommend raising it.

This is the dark factory Phase 3 pattern applied conservatively: automation expands based on evidence, not configuration.

Files:
- `internal/store/report.go` — quality metric aggregation (mean, P50, P90, override rate)
- `cmd/dashboard/` — quality trend charts

### Phase Q3: Reflective Health Monitor [DONE]

Implemented in PR #65 (2026-03-17). The `/health-check` POST endpoint queries three metrics from existing tables: confidence score trend (7-day vs all-time average), stuck agent sessions (non-terminal stage for >1 hour), and orphaned triage sessions (no bot_comment, not closed, >1 hour old). When thresholds are violated, the endpoint creates deduplicated `[Health Alert]` GitHub issues on the bot's own repo. The daily dashboard cron triggers health checks for each monitored repo. LLM response time P90 is deferred (not yet tracked in the database). See `docs/plans/2026-03-16-phase-q3-health-monitor.md` for the full plan.

Files:
- `internal/store/health.go` — health metrics queries and threshold evaluation
- `internal/store/health_test.go` — table-driven threshold tests
- `internal/github/client.go` — `SearchIssues` method for alert deduplication
- `cmd/server/main.go` — `/health-check` endpoint
- `.github/workflows/dashboard.yml` — daily health check step

### Phase Q4: Retrospective Compliance Audit

Once the audit log has 50+ entries, run a periodic job that re-evaluates past agent outputs with the current safety validators and quality judge. Flag any outputs that would fail today's checks. This catches two problems: safety standard drift (our validators got stricter but old outputs wouldn't pass) and model drift (Gemini's behavior changed and quality degraded).

Files:
- `cmd/audit/main.go` — retrospective audit tool
- `internal/store/agent.go` — query for completed sessions with stored outputs

## Stream 3: Communication and Framing

### How to Talk About This

The 2025 Stack Overflow survey found that developer trust in AI accuracy dropped from 40% to 29% year-over-year, even as usage reached 80%. The core tension: developers use AI tools but don't trust them. The projects that succeed frame AI as augmentation within existing workflows, not as a replacement or a separate paradigm.

For the triage bot, the framing principles are:

Lead with the problem, not the technology. "Checks if your bug report has the information maintainers need" resonates more than "uses Gemini embeddings with pgvector similarity search." Every feature description should start with the user's pain point: incomplete bug reports waste maintainer time, duplicate issues fragment discussion, enhancement requests lack technical context.

Name phases by what they do for the user. "Missing information check" not "Phase 1." "Known solutions" not "Phase 2." "Similar issues" not "Phase 3." "Related context" not "Phase 4a." These names should appear in the bot's comments, the dashboard, and any public documentation.

Frame safety as quality engineering. "Two-layer quality assurance" not "safety guardrails." "Holdout quality scoring" not "LLM output validation." The structural validator and LLM reviewer are signs of engineering maturity, not admissions that the system might go wrong. Renovate's confidence badges show that transparency about uncertainty builds trust.

Frame the agent's approval gates as human-reviewed research. "The Enhancement Researcher drafts analysis for maintainer review" positions the human as the decision-maker, not as a fallback when the bot fails. GitHub's Copilot framing is explicitly "developers will always own the merge button."

Make limitations specific and honest. "The bot doesn't detect duplicates that use very different wording" is more trustworthy than silence. Evil Martians' research shows that developers trust tools that are honest about what they can't do.

### Phase C1: User-Facing Documentation

Write a README section and a short guide that explain what the bot does in user-facing language. Include:

- What each phase does (in problem-first language)
- What the bot doesn't do (explicit limitations)
- How to give feedback (reaction + issue template)
- How the Enhancement Researcher works (for repos where it's enabled)
- A link to the dashboard

This is also where the confidence badges idea applies: the bot's comments could include a brief note about how confident the suggestions are, following Renovate's pattern.

### Phase C2: Public Changelog

Maintain a changelog in the repo (or as GitHub Releases) that documents each improvement in outcome-focused language. Evil Martians' research found that consistent update cadence is the strongest trust signal — regular updates, even small ones, signal momentum and reliability.

The changelog should describe what changed for users, not what code changed. "Duplicate detection now finds matches even when issues use different terminology" rather than "Updated embedding model to gemini-embedding-001 with 768 dimensions."

### Phase C3: Community Feedback Loop

Once the bot has been running on teams-for-linux for 30+ days with feedback metrics, publish a summary of what's working and what isn't. This could be a GitHub Discussion, a pinned issue, or a section in the README. Share the dashboard metrics publicly. Invite specific feedback on the phases that are underperforming.

The CNCF recommends this for contributor growth: publishing roadmaps and holding regular triage creates a sense of collective ownership. For a bot, the equivalent is showing users that their feedback directly influenced what got improved.

## Stream 4: Quick Wins

These are low-to-medium effort improvements identified from reviewing comparable projects (Project Oscar/gabyhelp, VS Code's triage classifier, CodeRabbit, Block's goose, GitHub Agentic Workflows) and analysing gaps in the current implementation. They can be executed independently of the other streams.

### Phase W1: Dashboard Links and Compliance Score [DONE]

Two small changes that improve the maintainer's daily workflow.

Dashboard shadow issues now link to shadow repos, comments link to GitHub comment anchors (PR #35). Shadow issue titles include a compliance score showing how many template sections are filled (e.g., `[Triage] [3/4] #123: Title`), making the lgtm/reject decision faster.

Files:
- `cmd/dashboard/template.html` — shadow issue and comment links
- `internal/webhook/handler.go` — compliance score in shadow issue titles (triage and retriage)

### Phase W2: Auto-Close Stale Shadow Issues [DONE]

Shadow issues that haven't received lgtm/reject within 14 days are now automatically closed with an expiry note. The `/cleanup` POST endpoint queries stale agent and triage sessions, posts a closure comment, closes the GitHub issue, and marks the session as complete (agent) or closed (triage). The daily dashboard workflow triggers cleanup via WIF-authenticated request to Cloud Run before generating dashboards.

Bug fix: added `closed_at` column to `triage_sessions` (migration 009) so that closed triage sessions aren't re-listed on subsequent cleanup runs.

Files:
- `migrations/009_triage_session_closed_at.sql` — adds `closed_at` column
- `internal/store/agent.go` — `ListStaleSessions` excludes closed triage sessions, `MarkTriageSessionClosed` added
- `cmd/server/main.go` — cleanup handler marks both agent and triage sessions
- `.github/workflows/dashboard.yml` — daily cleanup job via WIF + Cloud Run

### Phase W3: Time-to-First-Response Metric [DONE]

Track the time between issue creation and the first bot response (triage session creation). Displayed on the dashboard as "Avg Response Time" with human-readable formatting (seconds/minutes/hours).

To make this metric accurate, `UpsertIssue` now preserves the original GitHub issue creation timestamp when provided. The seed command passes `created_at` from the JSON index. A re-seed will backfill accurate timestamps for the 1,356 historical issues.

Files:
- `internal/store/report.go` — avg response time query added to dashboard stats
- `internal/store/postgres.go` — `UpsertIssue` accepts explicit `created_at`
- `cmd/dashboard/template.html` — response time card
- `cmd/seed/main.go` — passes `created_at` from issue JSON

### Phase W4: Retriage Command [DONE]

Allow the maintainer to comment `/retriage` on a public issue to force the bot to re-run its analysis. This is useful after a user edits their issue body (e.g., adding missing information that Phase 1 flagged) and follows Block's goose pattern of "AI doesn't act until the maintainer explicitly invokes it."

This also provides a way to demonstrate value on existing issues, not just new ones, which helps during the Stage A data collection period when new issue volume may be low.

Implementation: the webhook handler detects `/retriage` as a comment on a source repo issue (not shadow), re-runs all phases with the current issue body, and posts the result to a new `[Retriage]` shadow issue. The triage session is upserted so lgtm/reject still work.

Files:
- `internal/webhook/handler.go` — detect `/retriage` in issue comments, re-run pipeline via `handleRetriage`

### Phase W5: Backfill Quality Baseline [DONE]

One-time CLI tool that fetches the last N closed issues from the GitHub API, runs the full triage pipeline (phases 1-4b), and stores the results in `bot_comments`. Configurable via `BACKFILL_LIMIT` (default 50), `REPO`, and `SOURCE_REPO` env vars. Includes 1-second rate limiting between issues for Gemini free tier compliance.

Files:
- `cmd/backfill/main.go` — one-time backfill script
- Clean up after execution

## Stream 4: Strategic Intelligence (NEW — 3-month roadmap)

This stream transforms the project from a reactive triage bot into a proactive repository strategist. Full design: `docs/plans/2026-03-18-repository-strategist-design.md`. Implementation plan: `docs/plans/2026-03-18-repository-strategist-implementation.md` (21 tasks, 3 batches).

### Month 1: Institutional Memory Strengthening

Per-repo config file (`.github/butler.json`), event journal recording all repo activity (issues, PRs, pushes, releases), automatic document re-embedding on push, cross-reference index linking documents to issues and ADRs. New endpoints: `/ingest` (batch event ingestion from GitHub Actions), authenticated via `INGEST_SECRET`. New GitHub App permission: `contents: read` for config file loading. Shared ingest package extracted from seed CLI.

### Month 2: Pattern Detection Engine

Synthesis engine with `Synthesizer` interface and weekly cron via GitHub Actions. Three synthesizers: issue cluster detection (vector similarity grouping over recent issues, flags uncovered themes), decision drift detection (PR-vs-ADR contradiction, roadmap staleness), upstream impact analysis (cross-references new upstream releases against deferred ADRs and open issues). Weekly briefings posted as `[Briefing]` shadow issues.

### Month 3: Strategic Output (revised 2026-03-23)

Revised to avoid duplicating work that `repo-butler` (https://github.com/IsmaelMartinez/repo-butler) already handles. The guiding principle is collaborate, not compete: this bot owns institutional memory and analysis (vector store, embeddings, synthesis), while repo-butler owns presentation and action (dashboards, roadmap PRs, improvement proposals). The two integrate via `.github/butler.json` auto-discovery and the `/ingest` and `/report/trends` endpoints.

Delegated to repo-butler (dropped from this roadmap):
- Roadmap task generator (Task 16) — repo-butler's IDEATE+PROPOSE phases already generate LLM-powered improvement ideas as GitHub issues with priority sorting and approval gates. Instead, enrich `/report/trends` so repo-butler can use cluster findings as IDEATE input context.
- Monthly state-of-the-project briefing (Task 18) — repo-butler's ASSESS+REPORT phases already produce per-repo HTML dashboards with PR velocity, issue trends, release cadence, and weekly trend computation deployed to GitHub Pages. Instead, expose synthesis findings as structured JSON via `/report/trends` for repo-butler to incorporate.

Kept in this bot:
- ADR lifecycle management (Task 17): revision proposals when code consistently drifts from ADR, gap detection when no ADR covers an active area. This requires the vector store and embedding-based cross-referencing that repo-butler doesn't have.
- Enrich `/report/trends` endpoint (new): expose cluster findings, drift signals, and upstream impacts as structured JSON so repo-butler's ASSESS phase can incorporate them into richer reports and IDEATE can use them as context for proposals.
- Dashboard synthesis metrics (Task 19): add synthesis stats to the live `/dashboard` endpoint. This complements repo-butler's static HTML reports rather than competing — the live dashboard shows operational health while repo-butler shows portfolio trends.
- Multi-repo hardening and documentation (Task 20): config validation, butler.json schema docs, getting-started guide.
- LLM daily usage budget tracker (Task 21): DONE (PR #84, implemented as part of safety hardening).

### Positioning vs GitHub Agentic Workflows and Repo Butler

GitHub's Continuous AI pattern and Agentic Workflows (technical preview) will commoditise operational agents: doc sync, test coverage, CI triage, label management. Repo-butler occupies the planning and reporting layer: portfolio dashboards, roadmap PRs, improvement proposals. This bot occupies the institutional memory layer beneath both: vector store of ADRs, roadmap, and upstream docs; accumulated learning from maintainer feedback; temporal awareness via event journal enabling pattern detection over time. No stateless per-run agent can produce "this ADR has been contradicted by 3 PRs this month" or "Electron v40 just shipped the feature you deferred in ADR-007."

The three-layer stack: GitHub Agentic Workflows (operational chores) → repo-butler (planning, reporting, proposals) → triage bot (institutional memory, analysis, triage). Each layer feeds the one above via APIs and shared config.

## What We're Not Doing

Generic triage features — labeling, deduplication, routing — are explicitly out of scope. The competitive analysis (2026-03-16) confirmed these are commodity features. GitHub Agentic Workflows will further commoditise operational chores (doc sync, test coverage, CI triage). New feature work should be evaluated against this framing: if a proposed feature could be offered by any generic or stateless agent, it's likely not worth building here. The differentiator is institutional memory and strategic reasoning over time.

Roadmap proposals and portfolio-level reporting are delegated to repo-butler (2026-03-23). This bot provides the intelligence layer (synthesis findings, cluster detection, drift signals) via API endpoints; repo-butler consumes them and handles presentation, roadmap PRs, and improvement proposals. Do not duplicate planning or reporting features that repo-butler already provides.

Automated PR creation for code changes is out of scope for the current trust-building phase — the butler proposes, humans implement. Direct public-facing synthesis output is out of scope — all strategic intelligence goes through shadow repos. Integration with GitHub Agentic Workflows is deferred until GA.

Multi-agent orchestration, queue-based architecture, and digital twin testing remain premature. The triggers for reconsidering are parked in `docs/research/inspiration-repos.md`.

## Measuring Success

### Triage Pipeline (Streams 1-3)

After 90 days of running with feedback infrastructure:

- Phase 1 effectiveness: >50% of users edit their issue after a missing-info prompt
- Phase 2 effectiveness: >20% of issues where troubleshooting was suggested get self-resolved
- Agent quality: average holdout judge score >75, human override rate <30%
- User sentiment: reaction ratio (thumbs-up / total reactions) >70%
- Time-to-first-response: median time from issue opened to first useful response <5 minutes
- Shadow repo engagement: <30% of shadow issues expire without maintainer action
- Operational health: <5% comment posting failure rate, <1% safety check failures

### Strategic Intelligence (Stream 4)

After 90 days of the synthesis engine running:

- Briefing engagement: >50% of weekly briefings receive a maintainer reaction or reply within 48 hours
- Cluster detection precision: >60% of flagged issue clusters acknowledged as real patterns (not noise)
- Drift detection precision: >40% of ADR drift signals lead to an ADR revision or code correction
- Roadmap proposals: at least 2 accepted roadmap proposals in 90 days
- ADR proposals: at least 1 accepted ADR revision or new ADR in 90 days
- State-of-project usefulness: maintainer reads and shares the monthly briefing (qualitative)
- Multi-repo readiness: a second repo can onboard with <1 hour of setup time
- Cost: total Gemini API usage stays within free tier across all configured repos

These numbers are initial targets — the first 30 days of data will calibrate what's realistic.
